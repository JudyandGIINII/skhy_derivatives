# H1 과거 데이터 기반 백테스트 검증 방안 조사 및 설계 (Historical Backtest Feasibility)

본 문서는 KIS 실시간 웹소켓 특성상 과거 시세 재생이 불가능하여 발생하는 "실전 승격 전 6개월의 수집 대기 시간" 문제를 우회·보완하기 위한 과거 데이터 기반 검증 가능성(Feasibility) 및 사전 반증 방안을 정리합니다.

---

## 1. 과거 데이터 원천 조사 및 분석 (KRX, KIS, Naver/Yahoo)

연속장 OFI(Order Flow Imbalance), 실시간 프로그램 매매 proxy, 그리고 종가경매 outcome을 부분적으로라도 역사적으로 복원할 수 있는지 조사한 결과는 다음과 같습니다.

### 1.1. 한국거래소 (KRX) 정보데이터시스템
- **일별 종가경매 거래량**: 특정 종목의 일별 단일가 매매 거래량(15:20 ~ 15:30 체결 합계) 및 종가 단일가 가격 데이터는 무료로 조회 및 CSV 다운로드가 가능합니다. (근거 URL: [KRX 정보데이터시스템](https://data.krx.co.kr))
- **프로그램매매 일별 통계**: 일별 코스피/코스닥 시장 전체 및 종목별 프로그램 매매 순매수/매도 수량 및 금액 추이를 무료로 획득할 수 있습니다.
- **과거 분봉/호가 데이터 제공**: 10단계 호가잔량(Order Book) 및 분 단위 호가 데이터는 무료 웹 서비스 범위가 아닙니다. **KRX 데이터 마켓플레이스**에서 별도의 '유료 데이터 상품'으로 구매해야 하며, 기간과 범위에 따라 수십만 원에서 수백만 원의 비용 및 재배포 제한 라이선스가 적용됩니다. (근거 URL: [KRX Market Data](https://marketdata.krx.co.kr))

### 1.2. 한국투자증권 (KIS) Open API
- **과거 분봉 데이터**: `FHKST01010400` (국내주식 기간별 분봉 조회) API를 통해 최근 30일 이내의 분봉 데이터를 무료로 조회할 수 있습니다. (근거 URL: [KIS Developers 국내주식 시세](https://apiportal.koreainvestment.com))
- **과거 호가 데이터**: 과거 시점의 10단계 호가잔량(OFI 계산의 핵심)은 REST API를 통해 전혀 제공되지 않습니다. KIS는 실시간 웹소켓으로만 호가 데이터 스트림을 전송하므로, 과거 아카이브 구축은 오직 실시간 캡처로만 가능합니다.

### 1.3. Naver / Yahoo Finance (보조 원천)
- **Naver 금융**: 특정 종목의 일별 시세, 프로그램 매매 동향(순매수 수량/금액) 등의 데이터를 HTML 파싱을 통해 무료로 적재할 수 있습니다. (근거 URL: [네이버 페이 증권](https://finance.naver.com))
- **Yahoo Finance**: 과거 수개년 동안의 일별 시세(OHLCV) 및 수정 주가(Adjusted Close)를 무료 API(yfinance 등)로 대량 제공하지만, 국내 주식 특화 데이터인 프로그램 매매 세부 통계나 종가 단일가 거래량은 제공하지 않습니다. (근거 URL: [Yahoo Finance](https://finance.yahoo.com))

## 2. 검증 가능 영역 vs 라이브 수집 전용 영역의 분리

과거 데이터로 보완·검증할 수 있는 영역과 반드시 6개월간의 라이브 수집을 통해서만 캡처 가능한 영역을 명확히 구분합니다.

| 분류 | 과거 데이터로 검증 가능 (Historical Proxy) | 라이브 수집 전용 (Live-Only Raw) |
| :--- | :--- | :--- |
| **종가경매 (Close Auction)** | - 일별 최종 단일가 체결량 및 가격 변동<br>- 15:20 직전 가격 대비 15:30 최종 종가의 변동률 (Outcome) | - 15:20 ~ 15:30 사이 예상체결가/예상체결량의 10분간의 실시간 추이 및 변동 흐름 |
| **프로그램 매매 (Program Trading)** | - 일별 최종 프로그램 순매수/순매도 누적 금액/수량<br>- 과거 분봉 거래량을 기반으로 한 일중 분배 Proxy | - 실시간 프로그램 매매 체결 패킷(`H0STPGM0`)을 이용한 초/분 단위 실시간 유입 속도 및 강도 |
| **호가 불균형 (OFI)** | - 불가능 (과거 호가잔량 시계열 부재) | - 10단계 호가잔량 변화량(`H0STASP0`)을 실시간 계산한 연속장 일중 호가 불균형(OFI) 신호 생성 |
| **대체거래소 (Nextrade)** | - 불가능 (NXT 과거 데이터는 아카이브 부재) | - KIS 웹소켓(`H0NXPGM0`, `H0UNPGM0`)을 통한 실시간 호가/프로그램 유입량 수집 |

## 3. 과거 대용 검증 시 설계 및 가이드라인

과거 데이터로 프록시(Proxy) 백테스트를 수행할 때 발생할 수 있는 왜곡을 방지하기 위한 안전장치입니다.

### 3.1. 짧은 표본 과장 금지 (PRD 10.2 준수)
- KIS API 등을 통해 구한 과거 분봉 표본(예: 30일 또는 3개월)은 전체 모집단을 대변하기에 너무 짧습니다.
- 이 기간의 높은 Sharpe Ratio나 승률을 전략의 대표 성능으로 과장 보고해서는 안 되며, 반드시 **표본 기간이 짧음을 명시**하고 t-통계량 및 p-value 등 **통계적 유의성 검정을 병기**해야 합니다.

### 3.2. 룩어헤드 편향(Look-Ahead Bias) 원천 차단
- 일별 프로그램 매매 총량이나 당일 최종 종가를 일중 백테스트 계산 시점에 미리 참조(Look-ahead)하지 않도록 Point-in-Time 원칙을 고수합니다.
- 예: 15:00 백테스트 시그널 계산 시에는 15:00까지 누적된 거래 정보만 사용해야 하며, 당일 최종 프로그램 매매 규모나 15:30에 결정된 종가를 모델 입력값에 주입하는 등의 룩어헤드 데이터 누수를 코드 수준에서 철저히 격리해야 합니다.

### 3.3. 데이터 계보 (Data Lineage) 유지
- 과거 데이터로 시뮬레이션한 피처와 백테스트 결과는 라이브 캡처 데이터와 명확히 구분된 데이터베이스 스키마 및 파일 경로(`data/historical/...`)에 적재합니다.
- `raw_record_catalog` 등록 시 `source` 필드에 `KIS_HISTORICAL_REST` 혹은 `KRX_HISTORICAL_PORTAL` 등으로 명시하여 실전 라이브 데이터와의 Lineage 혼선을 원천 차단합니다.

## 4. 권고: 6개월 대기 단축을 위한 사전 반증(Pre-Falsification) 방안

라이브 데이터 6개월 수집 완료 전에, **무료 과거 데이터를 활용하여 전략의 가정을 사전에 검증하고 기각(Falsify)하는 절차**를 권고합니다.

### 4.1. 사전 반증 가설 설정
- **핵심 가정**: "장중(또는 15:20 직전까지) 프로그램 순매수가 대량 유입된 날에는, 종가 단일가 경매(15:20~15:30)에서 매수 압력이 지속되어 종가가 상승 마감하거나 종가 경매 거래량이 급증할 것이다."
- 만약 이 상관관계가 과거 데이터 상에서 전혀 관측되지 않는다면, 실시간 웹소켓 데이터를 6개월 동안 모아도 전략이 작동하지 않을 것이므로 **라이브 6개월을 기다릴 필요 없이 즉시 기각**이 가능합니다.

### 4.2. 실행 프로토콜
1. **데이터 확보**: KRX 정보데이터시스템 및 네이버 금융에서 지난 3~5년 동안의 SK하이닉스(`000660`) 일별 프로그램 매매 누적 순매수 대금, 15:20 가격(단일가 시작 전 직전 체결가), 15:30 최종 종가, 그리고 종가경매 거래량 데이터를 수집합니다.
2. **회귀분석 수행**:
   $$\Delta P_{\text{close}} = \beta_0 + \beta_1 \cdot (\text{ProgramNetBuy}) + \epsilon$$
   - 여기서 $\Delta P_{\text{close}}$ 는 15:20 가격 대비 15:30 종가의 수익률입니다.
3. **기각 기준**:
   - 회귀계수 $\beta_1$ 의 t-통계량이 1.96 이하(p-value > 0.05)이거나, 부호가 음수일 경우 본 전략의 핵심 가설은 **사전 반증(Falsified)**된 것으로 판정하여 프로젝트를 피벗하거나 개발을 전면 재검토합니다.
4. **효과**:
   - 이 반증 프로세스를 통과한 경우에만 6개월 라이브 데이터 캡처를 지속함으로써, 불필요한 시스템 운영 비용과 연구 리소스를 대폭 절감할 수 있습니다.

## 1. 기술 스택 선정 및 선정 근거

1인 연구 및 백테스팅/페이퍼트레이딩 환경의 특성을 반영하여 **가벼운 운영 비용, 데이터 재현성, 그리고 빠른 분석 피드백**을 극대화할 수 있는 스택으로 구성한다.

| 기술 요소 | 선택된 기술 | 선정 근거 |
| --- | --- | --- |
| **개발 언어** | Python 3.11+ | 금융 데이터 분석 생태계(Pandas, NumPy, Scipy)의 압도적 성숙도. KIS/Toss/KRX API 연동을 위한 공식/비공식 라이브러리 지원 풍부. |
| **데이터 저장소** | DuckDB + Parquet | 로컬 1인 분석 환경에서 최적의 성능을 발휘하는 OLAP 엔진. 불변의 원시 데이터를 압축률이 높은 Parquet 파일 형식으로 로컬에 적재하고, DuckDB의 인메모리/디스크 쿼리를 결합하여 틱(Tick) 및 분봉 데이터 정규화/조인을 서버 관리 오버헤드 없이 고속 처리. |
| **설정 및 비밀값 관리** | `python-dotenv` | API 키, 비밀값, 계좌번호 등을 환경변수로 분리하여 원격 저장소(`https://github.com/JudyandGIINII/skhy_derivatives`) 노출을 원천 방지. |
| **이벤트 루프 및 엔진** | Custom Event Loop | 단일 국가/단일 자산군 위주의 범용 백테스팅 라이브러리(예: Backtrader)는 다중 거래소(KRX, NXT, Nasdaq, HKEX), 다중 통화(KRW, USD, HKD), 다중 세션(프리, 정규, 애프터), 그리고 부분체결/시간 경과에 따른 한쪽 다리 청산 타임아웃을 유연하게 처리하기 어려움. 따라서 수동 제어가 가능한 순수 Python 기반의 이벤트 큐 기반 커스텀 백테스트 루프를 자작함. |
| **테스트 프레임워크** | `pytest` + `unittest.mock` | 공급자 계약 테스트 및 룩어헤드(Look-ahead) 방지, 데이터 품질 테스트를 작성하고 자동화하기 위한 업계 표준 프레임워크. |
| **리포트 및 UI** | Streamlit + Plotly | 분석 결과 시각화(수익률, MDD, 슬리피지 분포, 95% ES 등) 및 실시간 페이퍼 트레이딩 모니터링 화면(PRD 15장)을 최소한의 오버헤드로 고급스럽고 Premium하게 구축하기 위한 선택. |

## 4. 아키텍처 상세 설계

### 4.1 데이터 계층 (Data Layers)
원문 데이터 보존과 전략에 유용한 정보의 고속 추출을 보장하기 위해 데이터 흐름을 6개 계층으로 구조화한다.

```text
[Raw Layer: 불변 Parquet] 
       │  (정규화 ETL)
       ▼
[Normalized Layer: RecordEnvelope]
       │  (데이터 품질 검증)
       ▼
[Feature Layer: 괴리율, 리밸런싱 압력]
       │  (전략 평가)
       ▼
[Signal Layer: 진입 방향, 신뢰도]
       │  (리스크 엔진 필터링)
       ▼
[Order Layer: OrderIntent]
       │  (체결 모델 시뮬레이션)
       ▼
[Execution Layer: PaperFill]
```

1.  **Raw Layer (원시)**: API로 가져온 JSON/CSV 형태의 원문을 그대로 Parquet 파일로 보관하며, 변경이나 삭제가 허용되지 않는다.
2.  **Normalized Layer (정규화)**: 공통 데이터 계약 `RecordEnvelope` 구조에 매핑한다. 통화(`ISO 4217`), 타임스탬프(`UTC epoch nanoseconds`), 식별자(`instrument_id`)를 일치시키고 지연 및 데이터 품질 플래그를 표기한다.
3.  **Feature Layer (특징)**: ADR 프리미엄(중간값 및 실행 가능 가격), 스왑 기반 레버리지 상품의 예상 리밸런싱 수급액, 공통 반도체 팩터를 제거한 NXT/Nasdaq 개별 잔여수익 등을 계산하여 저장한다.
4.  **Signal Layer (신호)**: 전략별 시그널 객체를 생성한다. 여기에는 신호 발생 원천 데이터의 ID 리스트, 모델 버전, 설정값 해시가 반드시 동반된다.
5.  **Order Layer (주문)**: 포지션 사이징을 거친 페이퍼 주문 객체 `OrderIntent`를 포함한다.
6.  **Execution Layer (체결)**: 부분체결, 호가 깊이를 감안한 슬리피지, 매매 비용 등이 반영되어 확정된 `PaperFill` 객체이다.

### 4.2 공급자 포트 (Provider Ports)
구동 환경(백테스트/실시간)에 무관하게 동일한 비즈니스 로직을 사용하기 위해, 데이터 원천 및 브로커 엔드포인트를 추상 인터페이스로 정의한다.

```text
               +-----------------------------+
               |      Strategy / Engine      |
               +-----------------------------+
                              │
       ┌──────────────────────┼──────────────────────┐
       ▼                      ▼                      ▼
[MarketDataProvider]   [ReferenceDataProvider] [BrokerProvider]
       │                      │                      │
  ┌────┴────┐            ┌────┴────┐                 ▼
  │  KIS    │            │  KRX    │        [PaperBrokerProvider]
  │  Toss   │            │  발행사 │        (실주문 어댑터는 주입 금지)
  └─────────┘            └─────────┘
```

*   **`MarketDataProvider` (시세 공급 포트)**: 실시간 틱/호가 데이터를 구독하고 해제하는 책임을 진다. KIS(주), Toss(부) 구현체를 동적으로 바인딩한다.
*   **`ReferenceDataProvider` (기준정보 공급 포트)**: 종목 마스터, 기업행사(수정주가 계수), 보통주-ADR 비율 및 예탁기관 공시 상태 조회를 담당한다.
*   **`HistoricalDataProvider` (과거 데이터 공급 포트)**: 과거 기간의 바/틱 단위 통계를 수집하고 계보를 추적한다.
*   **`BrokerProvider` (브로커 포트)**: 계좌 상태, 포지션 조회 및 주문 처리를 대행한다. v1에서는 `PaperBrokerProvider`만 생성 및 주입하고, 실제 증권사 실주문 커넥터는 설계 수준에서 배제하여 실수에 의한 실거래 실행을 원천 봉쇄한다.

### 4.3 전략 플러그인 (Strategy Plugins)
*   모든 전략은 추상 클래스 `StrategyPlugin`을 상속하여 작성한다.
*   전략 객체는 공급자 포트에서 올라오는 `MarketQuote` 또는 `Timer` 이벤트가 트리거될 때 `on_market_data()` 메서드를 통해 입력 데이터를 수신한다.
*   계산 완료 후 의사결정이 필요할 경우 공통 `Signal` 객체를 반환하고, 거래 기회가 없거나 리스크 조건 미충족 시 `null`을 반환한다.

### 4.4 이벤트 기반 백테스트 엔진 (Event-Driven Backtest Engine)
*   과거 틱/분봉 데이터를 시간 순서대로 정렬하여 큐(Queue)에 적재하고 하나씩 꺼내어 처리하는 순수 이벤트 루프 형태로 구현한다.
*   동일한 타임스탬프 내의 이벤트들은 `거래소 시세 이벤트 -> 전략 계산 -> 리스크 엔진 검증 -> 주문 생성 -> 체결 처리` 순으로 엄격히 동기적 정렬 규칙을 따른다.
*   단일 난수 시드와 결정론적 상태 전이를 통해 동일 데이터와 설정에 대해 100% 동일한 결과(재현성)를 얻도록 설계한다.

### 4.5 페이퍼 브로커 (Paper Broker)
*   모의 계좌를 구동하기 위한 가상 원장이다.
*   주문 지연 시간(Latency) 및 지정가 터치 규칙을 반영하고, 매수/매도 호가의 잔량을 검사하여 부분 체결을 처리하는 시뮬레이션 모델을 장착한다.
*   원화(KRW), 달러(USD), 홍콩달러(HKD) 다중 통화 지갑 잔고를 별도로 보존하고 일일 환율에 따른 마진(Margin)을 산출한다.

### 4.6 리스크 엔진 (Risk Engine)
*   `RiskEngine`은 전략이 전송한 `OrderIntent`를 최종 승인하기 전 가로채어 검사한다.
*   **검사 제약 사항**:
    1.  **자산 한도**: 거래당 최대 위험은 총 자산의 0.25%, 일일 최대 누적 손실은 1.0%를 넘지 못하게 수량을 차단/축소(`REDUCE`)한다.
    2.  **데이터 신선도**: 호가 나이(H1: 2초, H2/H3: 5초)를 초과해 지연된 시세 기반 주문은 즉각 `BLOCK`한다.
    3.  **킬스위치**: `ConversionStatus`가 `UNKNOWN` 또는 `SUSPENDED`이거나, 대차가 불가능하거나, 누적 MDD가 5.0%를 넘어서면 모든 신규 주문을 차단하고 킬스위치 상태로 전이한다.
*   판정 결과인 `RiskDecision`을 데이터베이스에 기록하여 추후 감사(Audit) 시 활용할 수 있게 한다.

## 5. 테스트 전략 (PRD 14장 반영)

시스템의 재현성과 안전성을 담보하기 위해 모든 핵심 계층에 정제된 Mock Fixture 기반의 자동화된 테스트를 작성한다.

### 5.1 공급자 계약 테스트 (14.1 반영)
*   **목적**: 외부 증권사 API 스펙 변경 및 네트워크 장애 대응력을 확인한다.
*   **테스트 케이스**:
    *   API Key 만료 및 인증 에러 상황 Mocking
    *   호출 제한(Rate limit) 도달 시 지수 백오프(Exponential Backoff) 및 Jitter 처리 확인
    *   WebSocket 연결 해제 시 재연결 및 REST API를 통한 누락 데이터 복구 시나리오 재생
    *   외부 API 응답에 담긴 계좌 암호 및 토큰이 로그 파일에 노출되지 않고 별표(`*`) 마스킹 처리되는지 검증

### 5.2 데이터 품질 및 정규화 테스트 (14.2 반영)
*   **목적**: 전략 계산의 입력값이 되는 시세 원장의 무결성을 확인한다.
*   **테스트 케이스**:
    *   중복 패킷, 시간 역순 패킷 유입 시 정렬/제거 로직 검사
    *   매수호가(Bid)가 매도호가(Ask)보다 높아지는 비정상 데이터 필터링 테스트
    *   환율 방향이 올바른지(`USD/KRW` 환산 비율 검사) 및 ADR의 보통주 10:1 매핑 연산 검증
    *   한국 장 종료 이후 미국의 실시간 호가와 한국의 이전 종가를 비교할 때 `stale_reference` 플래그가 주입되고 거래 대상에서 제외되는지 검증

### 5.3 전략 및 룩어헤드 방지 테스트 (14.3 반영)
*   **목적**: 모델링 오차 및 미래 데이터 참조 오류로 인한 백테스트 과장 오류를 잡아낸다.
*   **테스트 케이스**:
    *   **H1**: 15:10 신호 생성기에 15:20 종가 데이터 및 15:10 이후 공개되는 당일 NAV/AUM 수치를 강제 주입했을 때 예외가 발생하는지 확인(룩어헤드 차단 테스트)
    *   **H2**: `ConversionStatus`가 `SUSPENDED`이거나 대주 재고가 0인 상황에서 주문 신호가 생성되지 않고 `BLOCK` 처리되는지 확인
    *   **H3**: DST 서머타임 전환 경계일의 시간대 계산 및 한미 휴일 비일치일의 동시 세션 판정 유닛 테스트

### 5.4 체결 및 리스크 테스트 (14.4 반영)
*   **목적**: 최악의 상황에서도 시스템의 하방이 리스크 엔진에 의해 통제되는지 검증한다.
*   **테스트 케이스**:
    *   지정가 주문 미도달 시 미체결 유지 및 한쪽 다리만 체결된 상태에서 타임아웃 경과 시 자동 반대매매 청산 시뮬레이션
    *   모의 계좌 누적 MDD 5.0% 초과 상황을 강제로 연출하여 실시간 주문 생성 킬스위치 작동 확인
    *   스프레드 2배 확대 스트레스 및 주문 가능 거래량 반감 상황에서의 수익 성능 저하 측정

## 6. G-01~G-08 미결정사항의 처리 방안과 착수 순서

PRD 19장에 기술된 8가지 미결정사항(`G-01~G-08`)을 해결하기 위해, 각 항목에 대한 확인용 증거 기준과 처리 알고리즘을 다음과 같이 정의한다.

### 6.1 미결정사항 처리 방안

*   **`G-01` (보통주-ADR 전환 가능 여부)**: 회사 및 KSD(한국예탁결제원) 공시 사이트, 씨티은행 예탁 사이트를 크롤링/조사하여 실제 개인 고객을 대상으로 업무를 개시했는지 확인하고, 증거 수집 실패 시 `ConversionStatus = UNKNOWN`으로 설정하여 H2 거래를 전면 차단한다.
*   **`G-02` (API 한도 및 Token 수명)**: KIS 및 Toss 개발자 센터 스펙 문서를 바탕으로 Capability Probe 테스트 스크립트를 작성해 실측 지연과 호출 한도를 저장하며, API 제한 초과 시 임시 대기가 아닌 명시적 오류를 반환하게 만든다.
*   **`G-03` (종가 예상체결 및 프로그램 수집)**: KIS/Toss 실시간 호가 채널의 실데이터 패킷 구조를 조사하여 프로그램매매 순매수 및 장마감 10분전 예상 체결가/체결량 필드가 제공되는지 대조한다. 정보가 유실된 경우 H1 모델에서 해당 필드를 0으로 우회하지 않고, 이를 변수에서 제외한 **축소 모델(Reduced Model) 버전을 활성화**하고 품질 경고 플래그를 로깅한다.
*   **`G-04` (레버리지 상품 목록 및 PCF)**: KRX Open API의 종목 검색 엔드포인트를 사용해 "SK하이닉스" 및 2배/인버스 키워드를 자동 탐색하는 동적 종목 매퍼(Dynamic Instrument Mapper)를 설계한다.
*   **`G-05` (대차 잔고 및 금리)**: 브로커 포트에 대차 가능 수량 조회를 위한 Mock API를 선제 구축하며, 실시간 숏 재고를 입증할 수 없을 경우 리스크 엔진에서 H2의 숏 포지션 진입 시그널을 기각하도록 로직을 강제한다.
*   **`G-06` (데이터 라이선스 범위)**: 각 API 제공사와 맺은 이용약관을 참조하여, 로컬 내부 연구 목적으로만 Parquet 영구 보관을 허용하고 외부 재배포나 타인 양도는 차단하는 내부 보안 감사 규칙을 확립한다.
*   **`G-07` (규제 및 과세 비용)**: 국내 세법상 주식거래세(0.15% 내외), 미국 주식 양도세(22%), 배당소득세 및 금융소득 종합과세 기준과 공매도 관련 한시적 행정명령 유무를 체크하여 비용 모델의 파라미터 값으로 주입한다.
*   **`G-08` (페이퍼 자본 및 주문 규모)**: 초기 모의자본 1억 원 설정 하에, 주문 1건당 집행 규모는 해당 호가의 직전 1분 평균 5단계 호가 잔량 총합의 10% 또는 리스크 엔진이 산출한 1회 거래 최대 위험 한도(0.25%) 중 최솟값(가장 보수적인 값)으로 가속 제한하는 수식을 구현한다.

### 6.2 구현 착수 순서 (Dependency Chain)

선행 지식이 확정되어야 후속 설계 및 구현을 할 수 있으므로, 미결정사항의 의존성을 고려하여 다음과 같은 단계적 순서로 착수한다.

1.  **[G-02] KIS/Toss API Capability Probe**
    *   *이유*: 공급자 API의 물리적 한계와 성능을 검증해야 데이터 수집 포트 및 DTO 계약을 완성할 수 있음.
2.  **[G-04] KRX 레버리지 종목 동적 발견 모델링**
    *   *이유*: H1 전략에서 다룰 기초 자산 유니버스의 대상을 규정하고 DB 마스터 구조를 확립하기 위함.
3.  **[G-03] 종가 예상체결 및 프로그램매매 수집 필드 확인**
    *   *이유*: H1 전략의 수학적 압력 계산기에 실제로 주입 가능한 설명변수가 무엇인지 모델 구조를 조기 확정하기 위함.
4.  **[G-01] 보통주-ADR 전환 상태 및 비용 조사**
    *   *이유*: H2 전략이 실제로 이론적 무위험 차익거래 영역으로 작동할 수 있는 예탁원 채널 유효성을 확인하기 위함.
5.  **[G-05] 대차 가능 수량 및 비용 조사**
    *   *이유*: H2의 숏 다리 실행이 불가능할 경우 옵션이나 인버스 ETF를 활용한 대안 전략 구현을 설계하기 위함.
6.  **[G-07] 공매도/해외주식 세금 및 규제 비용 요건 수식화**
    *   *이유*: 백테스트 엔진에 주입할 실질 거래비용 파라미터(수수료, 세금)의 정확한 기준치를 잡기 위함.
7.  **[G-08] 페이퍼 자본 및 슬리피지/체결 모델 제약 설정**
    *   *이유*: 백테스트 엔진과 페이퍼 브로커의 체결 알고리즘 매개변수를 최종 튜닝하기 위함.
8.  **[G-06] 데이터 보관 정책 및 라이선스 고지**
    *   *이유*: 최종 가동 전 외부 데이터 활용 라이선스 위반 소지가 없는지 확인하고 로컬 적재 위치를 잠그기 위함.
